ISLAND FORGE - TALENT NODE SYSTEM REFERENCE
==========================================

Purpose
-------
This file is a planning handoff for the separate talent-system chat.
Do not treat this as final implementation. It defines the shape, domains, and placeholders.

Research scan notes
-------------------
- Large shared trees work when nodes have different weights: small passives, route nodes, notables,
  keystones, and specialized end choices.
- Grid-style progression works when players can see paths, spend points gradually, and unlock
  different identity routes without changing class.
- Multi-tree systems work when broad identity and specialized identity are separated.

Sources checked for structure inspiration:
- Path of Exile passive tree: dense shared network, class start positions, efficient pathing,
  stronger specialized bonuses.
- Final Fantasy X Sphere Grid: interconnected nodes, stat and ability unlocks, player-chosen route.
- World of Warcraft Dragonflight talents: two-branch talent revamp pattern.

Design goal
-----------
Every talent type should be able to support a full 100-node tree.
Each tree should touch the four main places:
- Forge: crafting, source routes, slot rolls, item optimization.
- Inventory: materials, cosmetics, foraging, storage.
- Talents: identity, long-term progression, keystones.
- Marketplace: trading, pricing, listing bonuses, rare exchange hooks.

100-node tree skeleton per talent type
--------------------------------------
Use this exact distribution for every major talent domain:

- 1 root node
  - The identity entry point for the domain.
- 36 minor nodes
  - Small passive bonuses, reliable incremental gains.
- 16 route/gate nodes
  - Branch connectors, island-zone unlocks, prerequisite pivots.
- 12 utility nodes
  - Quality-of-life, cooldown, stability, scouting, risk, or UI-facing bonuses.
- 10 crafting nodes
  - Forge output, slot control, roll quality, source-route bonuses.
- 8 exploration nodes
  - Foraging, mystery event, island travel, hazard mitigation.
- 6 cosmetic nodes
  - Cosmetic discovery, dye themes, profile flair, aura/skin unlock chances.
- 4 marketplace nodes
  - Trading fee, sell price, listing visibility, barter advantages.
- 5 notable nodes
  - Strong named bonuses that define a sub-build.
- 1 keystone node
  - Major identity tradeoff. Powerful, but changes how the domain plays.
- 1 capstone node
  - End-of-tree fantasy. Requires deep investment.

Total: 100 nodes per talent type.

Node type rules
---------------
- Minor: numeric, low-risk, repeatable structure.
- Utility: changes friction, pacing, cooldowns, or risk.
- Crafting: interacts with forge outcomes and artifact creation.
- Exploration: interacts with island material and cosmetic gathering.
- Marketplace: interacts with trade and economy.
- Notable: named build identity without severe downside.
- Keystone: major upside plus meaningful downside or routing commitment.
- Capstone: final fantasy for the domain, no more than one per 100-node tree.

Implementation notes for later chat
-----------------------------------
- Keep node data out of `cli_play.py` once it grows.
- Suggested future files:
  - `talents.py` for data model and unlock/evaluation logic.
  - `talent_data.py` or JSON/YAML for all domain/node definitions.
  - `talent_balance_reference.txt` for tuning rules and formulas.
- Store player-owned talents as:
  - `talents: {node_id: level}`
  - `talent_points`
  - `unlocked_domains`
  - `active_keystone_by_domain`
- Do not implement all 100 nodes per domain immediately. Use the 56 placeholders above as seed data,
  then expand one domain at a time.
